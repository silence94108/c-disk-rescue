## 3. 重点设计 B:自选搬家(B2,用户点名)

> 需求一句话:体检后把 C 盘里**大的文件夹摆出来、标明用途,用户手动选择是否迁移**——从"知识库命中才可搬"(现状 6 条 migrate 规则)扩展为"知识库推荐 + 受控自选"双轨。

### 3.1 与红线的关系(先说清)

- 红线原文是"**未知目录只展示、不提供删除按钮**"(需求文档 §1.3)。迁移不是删除:数据全程存在(复制→双校验→源改名 `.bak`→建链→用户确认后才删备份,`migrator.rs` 事务流程),失败自动回滚,事后可一键搬回。风险等级天然低一档。
- 需求文档 §3.3 边界为自选场景**预留了启发式**("含 `.exe` 不迁"仅用于未来开放用户自选目录),本设计启用它。
- `migrator.rs:319` 的防线原则"**接口形状即安全边界**——不接受前端传路径"**继续成立**:自选项复用孤儿 profile 的白名单模式(`scan.rs` 的 `ScanState.orphans` 先例)——后端生成候选并缓存原始路径,前端只回传候选 id。

### 3.2 候选发现(零成本,从扫描树筛)

数据源:`ScanState.result` 的目录 arena(大小、文件数、isReparse 全有),体检完成即可生成,不再碰磁盘。

**准入根**(第一版,收窄起步):

- `%LOCALAPPDATA%`、`%APPDATA%`、`%USERPROFILE%\AppData\LocalLow`
- `%USERPROFILE%\Documents` 的一级子目录(软件数据常驻区,如各类 `xxx Files`)

**第一版不含 `ProgramData`**:`fs::copy`(CopyFileExW)保留内容/时间戳/ADS 但**不保留源 ACL**,目标文件权限继承目标目录。用户目录内全是同一用户的 ACL 环境,无此问题;ProgramData 下服务数据搬过去可能出现服务账户读不了的权限事故,留待验证后再开。

**黑名单(硬编码否决,命中即不出现在候选)**:

| 目录 | 理由 |
|------|------|
| `AppData\Local\Microsoft`、`AppData\Roaming\Microsoft` | shell 核心数据(UsrClass.dat、开始菜单);微软系更新机制激进,junction 兼容风险最高;Edge 缓存已有 clean 规则覆盖 |
| `AppData\Local\Packages` | UWP/商店应用数据,特殊 ACL 与完整性要求,搬了商店应用崩 |
| `AppData\Local\Temp` | 是清理对象,不是迁移对象 |
| `AppData\Local\Programs` | 软件本体所在(VSCode/Chrome 用户级安装),竞品分析 §6 负面清单"软件本体搬家有意不做" |
| 已是 junction/symlink 的目录 | arena 的 `isReparse` 直接滤除(含本工具已迁移项) |
| `migrate-history.json` 已记录的 src | 已搬过 |
| OneDrive 根及其下 | 按需文件是 placeholder,复制会触发全量下载(需求文档 F1 特殊处理②);引擎层 `count_tree` 遇 reparse 也会拒(`migrator.rs:160-179`),候选层直接不展示,不让用户点了才报错 |
| 知识库 migrate 规则已覆盖的路径 | 归并进「推荐搬家」区,不在自选区重复出现 |

**exe 启发式(需求文档预留条款的落地,建议策略,待拍板)**:

- 目录**两级以内**出现 `.exe` → 判为"程序本体形态",不给搬家按钮,卡片显示"这个文件夹里有程序本体,搬走容易出问题,先不支持";
- 仅**深层**存在 `.exe`(如 `node_modules` 里的 esbuild.exe、游戏 mod 内容)→ 允许搬,徽标标"谨慎"。
- 依据:一刀切"含 exe 不迁"会误伤开发缓存与游戏资源这两类最大的自选空间来源;而"两级内含 exe"能准确圈住 `Programs\XXX\xxx.exe` 这种安装形态。exe 分布在扫描时顺手统计(与大文件收集同一趟,零额外成本)。

**阈值与粒度**:

- 默认 ≥1GB 进候选,按大小倒序,最多 30 项(避免决策疲劳;阈值落地时可调)。
- 粒度算法:候选从准入根的一级子目录(厂商层)起;若某候选 ≥2GB 且其最大子目录占比 ≥60%,下钻一层展示(最多下钻两层)。效果:`Tencent`(20GB)拆成 `WXWork`(15GB)+ 其他;单一产品目录不拆。参数落地时按真机效果调。

### 3.3 用途标注(三层递降)

1. **知识库命中**:直接用规则的 `displayName`/`explain`(现有 21 条)。
2. **厂商/产品词典**:独立小词典模块,与孤儿 profile 的 KNOWN 表(`scan.rs` 的 `orphan_hints`)合并共用数据源。首批方向(~30 词条):Tencent→腾讯系、Google、NetEase\CloudMusic→网易云音乐、JetBrains→开发工具、npm/pnpm/yarn→前端开发依赖、Steam→游戏平台、Epic、miHoYo→米哈游游戏、Docker、Unity、Adobe、Kingsoft→WPS、baidu→百度网盘、Thunder→迅雷、DingTalk、Feishu→飞书、NVIDIA→显卡驱动缓存……**与竞品分析 §5.2 特征库流水线是同一次调研的两个产出**:一次 Process Monitor 观察,规则库(清理/推荐迁移)与标注词典(自选区)同时受益。
3. **兜底**:原名展示 +"用途不明——点「打开位置」看看,不认识的先别动"。兜底项默认排在列表末尾。

### 3.4 契约改造清单(工程量的准确来源)

| # | 改动 | 说明 |
|---|------|------|
| 1 | `scan.rs` 新增 `get_migrate_candidates` command | 从 arena 筛候选,写 `ScanState.candidates: Mutex<HashMap<String, PathBuf>>`(候选 id → 原始路径白名单);返回展示用 DTO |
| 2 | `migrator.rs` 入口解耦 | 抽 `resolve_source`:`rule_id` 走知识库(现状),`pick:` 前缀走候选缓存;**事务流程(复制→双校验→改名→建链→恢复)一行不动** |
| 3 | `request_close` 同步解耦 | 现按 rule 查路径(`migrator.rs:575-581`),补候选 id 分支,同一优雅关闭逻辑 |
| 4 | `MigrateRecord.rule_id` 存合成 id | `pick:<路径哈希>`;`recover_pending_migration` 已有 rule_id 为空串的先例(`migrator.rs:533`),历史/还原/确认均按记录字段工作,天然兼容 |
| 5 | `types.ts` + `api/index.ts` | `MigrateCandidate` 接口与封装 |
| 6 | `Migrate.vue` 列表分区 | 三区:「推荐搬家」(知识库,现状)/「大文件夹自选」(候选,默认全不勾)/「官方搬法」(§3.6 引导卡);**向导状态机(waiting/moving/done)与占用轮询原样复用** |

### 3.5 防线小结(六道)

① 准入根收窄(仅用户目录数据区)→ ② 黑名单否决 → ③ exe 启发式 → ④ 候选缓存白名单(前端不传路径)→ ⑤ Restart Manager 占用检测 + 目标盘 NTFS/空间三重校验(现状)→ ⑥ 事务引擎本身(源数据全程存在、双校验、自动回滚、一键搬回)。

与磨针"负载均衡"的差异化正好落在这:它批量黑箱硬搬(用户评论"不知道转移了什么"),本项目**逐项标明用途、默认不勾、可撤回**——把竞品分析 §7 的定位("信任壁垒")做实。

### 3.6 Known Folder 官方搬法引导卡

`Downloads`/`Videos`/`Desktop`/`Pictures`/`Music`/`Documents` **本体**不进自选(有官方"位置"重定向机制,junction 是错误工具:OneDrive、库、备份工具都可能出边角问题),但它们往往正是最大的文件夹。处理:体检发现某 Known Folder ≥5GB 时,在搬家页尾部出**引导卡**(`action: "guide"` 枚举的首次落地,设计规范 §7 已定义):

> 「下载」文件夹有 23GB。Windows 自带官方搬家功能,比本工具更适合搬它:右键「下载」→「属性」→「位置」→「移动」,选到 D 盘即可。

交互模式与 guideOnly 清理项一致(蓝色徽标、教手动操作、不代做)。

### 3.7 验收

- 真机(本机)候选列表:知识库外的 ≥1GB 目录全部出现且用途标注命中率 ≥70%;`Microsoft`/`Packages`/`Programs`/`Temp`/已迁移项 0 出现。
- 自选搬一个 ≥2GB 的非知识库目录(如 JetBrains 缓存):软件重启正常、一键搬回正常、断电恢复测试(手动 kill 进程)回到原状。
- `pick:` 项在已搬家管理页与知识库项行为一致(确认删备份 / 搬回)。

## 4. 深度清理包(B3/B4,竞品分析 §5.1 的逐项工程评估)

| 子项 | 实现 | 评估 | 批次 |
|------|------|------|------|
| WinSxS | 子进程跑 `Dism.exe /Online /Cleanup-Image /StartComponentCleanup`,解析 stdout 百分比转进度事件 | ✅ 系统自判可清部分,安全;**不带 `/ResetBase`**(会锁死已装更新的卸载能力);耗时 5~20 分钟 → UI 上独立「深度清理」区,**不进一键优化**(一键优化的承诺是分钟级) | B3 |

| 休眠文件 | 大文件页 `hiberfil.sys` 卡片的"不能删"提示升级为「想腾这块?关闭休眠功能 ›」+ 代执行 `powercfg /h off`(needsAdmin);设置页放开关可逆(`/h on`) | ✅ 本质是设置切换不是清理;explain 必须写清代价:失去休眠与快速启动 | B3 |
| 系统还原点 | `vssadmin delete shadows /for=C: /oldest` 循环删旧、**永远保留最新一个**;caution 级默认不勾 | ✅ 可行;因不可逆且价值判断因人而异,放最后一批 | B4 |
| Windows.old | — | ⚠️ **建议维持 guideOnly**,见下 | 不排 |

**§4.5 Windows.old 为何不代删**:竞品分析 §5.1 提"接入系统清理接口",评估后不建议:cleanmgr 自动化(`/sageset`+`/sagerun`)是全局配置,会连带执行该编号下用户勾过的其它 handler,行为不可控;DISM 无删除 Windows.old 的稳定公开参数;自行遍历删除需处理 TrustedInstaller ACL(改系统目录所有权),恰是磨针"教用户改 system 权限"式的反面教材(竞品分析 §6 负面清单)。现状的 guideOnly 指引(磁盘清理勾「以前的 Windows 安装文件」)已是该场景的最优解。
